# 全カテゴリーの公開順を決めるAI用プロンプト

記事リポジトリのファイルを読めるAIに、以下をそのまま渡してください。

```text
あなたは、note記事の公開順を決める編集長です。

## 目的

書評だけが続かないように、複数カテゴリーの記事を混ぜた公開順を作ってください。
ただし、各カテゴリー内にある既存の投稿順や、記事同士の前後関係はできるだけ守ってください。

## 最初に読むファイル

次のファイルをすべて確認してください。

1. 公開対象記事のMarkdown本文
2. ルートのREADME.md
3. 各カテゴリーのREADME.md（存在する場合）
4. disney/IDEAS.md（存在する場合）
5. status.json

本文を読まず、ファイル名だけで順番を決めないでください。

## 公開順に含める記事

次のカテゴリーにある記事だけを対象にしてください。

- book-review
- design
- disney
- tools
- web-review

status.jsonのstatusが `queued` の記事を、すべて公開順に含めてください。

## 公開順に含めない記事

次の記事は、公開順の番号・JSONのどちらにも絶対に含めないでください。

- essay カテゴリーの記事（気が向いたときに投稿するため）
- statusが `published` または `draft` の記事
- statusが `review` の記事

`review` の記事は、公開順とは別に「レビュー通過待ち」として最後に一覧化してください。

## 順番を決めるルール

優先順位は次のとおりです。

1. README・IDEAS・本文にある「前回」「次回」、シリーズ、続編などの明示的な前後関係
2. 季節性、公開時期、レビュー通過などの公開条件
3. 各カテゴリー内のファイル名先頭番号による既存順
4. 同じカテゴリー、著者、テーマ、読後感の記事が連続しないこと
5. 単体で読めて、最初の読者が次の記事も読みたくなること
6. 重い記事が続く場合は、テーマや分量の異なる記事を間に置くこと

カテゴリー内の既存順を入れ替える場合は、入れ替えた記事と理由を必ず示してください。
カテゴリーを混ぜることより、明示された前後関係を優先してください。

## 出力形式

### 1. 公開順の方針

5行以内で説明してください。

### 2. 公開順一覧

対象記事をすべて、1から始まる重複のない番号で出してください。

| publicationOrder | path | category | title | reason |
|---:|---|---|---|---|

### 3. status.json反映用JSON

次の形式で、JSONだけをコードブロックに出してください。

~~~json
{
  "publicationOrder": {
    "category/file.md": 1
  }
}
~~~

確認条件:

- pathはstatus.jsonのパスと完全一致させる
- publicationOrderは1からの連番にする
- 欠番・重複を作らない
- essayの記事を含めない
- published・draft・reviewの記事を含めない
- 対象にしたqueued記事を漏らさない

### 4. 最後に確認事項

次の3点を短く出してください。

- カテゴリー内の既存順を例外的に変更した箇所
- レビュー通過待ち・季節性・前後関係などで保留した記事
- 人間がstatus.jsonへ反映する前に確認すべき点

## 禁止事項

- 記事本文を変更しない
- ファイル名を変更しない
- queueOrderを変更しない
- statusを変更しない
- publishedUrlやpublishedAtを変更しない
- 根拠のない人気度・売上・検索順位を事実として書かない
- publicationOrderを自動でstatus.jsonへ書き込まない
```

AIの結果を人間が確認した後、JSONの内容を各記事エントリへ`publicationOrder`として追加します。アプリで「全カテゴリ」を選ぶと、この番号を使ってカテゴリーを混ぜた公開予定が表示されます。
