# 全カテゴリの公開順を決めるAI用プロンプト

以下を、記事リポジトリのファイルを読めるAIにそのまま渡してください。AIには記事本文・各カテゴリのREADME・`disney/IDEAS.md`・`status.json`を参照させます。

```text
あなたは、noteで公開する記事群の編集長です。

目的:
複数カテゴリーの記事を混ぜ、読者が飽きず、各記事の理解が深まり、継続して読まれやすい公開順を決めてください。

必ず読むもの:
- 記事リポジトリ内の公開対象記事（Markdown本文）
- ルート README.md
- 各カテゴリーのREADME.md（存在する場合）
- disney/IDEAS.md（存在する場合）
- status.json

対象と除外:
- status.jsonで status が "queued" の記事を公開順の対象にする。
- "published" と "draft" は対象外にする。
- "review" はレビュー通過まで公開できないため、公開順の番号は付けず、最後に別一覧で示す。
- READMEやIDEASに書かれた公開順、シリーズ順、本文中の「前回」「次回」などの明示的な依存関係を確認する。

編集方針:
1. カテゴリーが連続しすぎないようにし、book-review、design、disney、essay、tools、web-reviewを可能な範囲で混ぜる。
2. 各カテゴリー内のファイル名先頭の番号は、そのカテゴリー内での既存の優先順を示す。原則としてこの相対順を維持する。
3. ただし、本文・README・IDEASに明示されたシリーズの続き、前後関係、季節性、事実確認待ち、レビュー待ちを優先する。
4. 同じ著者・同じテーマ・似た読後感の記事を連続させない。
5. 冒頭は新規読者にも内容が伝わりやすく、単体で読めて、次も読みたくなる記事にする。
6. 重い記事が続く場合は、短い記事・エッセイ・テーマの違う記事を間に挟む。
7. 書評だけが続く現状を解消する。ただし、カテゴリーを混ぜるために記事の前後関係を壊してはいけない。
8. 判断に迷う場合は勝手に同率にせず、どちらを先にしたか理由を書く。

出力:
最初に、全体方針を5行以内で説明する。

次に、status.jsonの "queued" 記事をすべて、1から始まる重複のない publicationOrder として、次の表で出力する。

| publicationOrder | path | category | title | reason |
|---:|---|---|---|---|

その後に、次のJSONだけをコードブロックで出力する。pathはstatus.jsonにある文字列と完全一致させ、publicationOrderは1からの連番にする。

{
  "publicationOrder": {
    "category/file.md": 1
  }
}

最後に、次の3つを短く出力する。
- 既存のカテゴリー内順を例外的に変更した箇所
- 依存関係・レビュー待ち・季節性などで保留した記事
- この順番をstatus.jsonへ反映する前に人間が確認すべき点

禁止事項:
- 記事本文、ファイル名、queueOrder、status、publishedUrl、publishedAtを変更しない。
- publicationOrderの欠番、重複、対象外記事の混入をしない。
- 本文を読まず、ファイル名だけで順番を決めない。
- 根拠のない人気度・売上・検索順位などを事実として書かない。
```

AIが返したJSONを人間が確認した後、`status.json`の該当記事エントリへ `publicationOrder` を追加します。アプリの「全カテゴリ」公開スケジュールは、この番号を優先してカテゴリーを混ぜて表示します。
