# AI画像生成連携の方針

このアプリでは、画像生成APIをGitHub PagesのJavaScriptから直接呼び出しません。現在採用する方式は「アプリでプロンプトを作る → 利用者が生成サービスで生成する → 画像を記事詳細から登録する」です。

## 採用理由

- 静的PWAに画像生成サービスの秘密鍵を置くと、利用者の端末や公開JavaScriptから取り出せる
- Disneyなど実在施設を扱う記事では、写真・ロゴ・キャラクター・来園者の肖像権や、生成画像を実写と誤認させる問題を利用者が確認する必要がある
- 生成APIはモデル、料金、サイズ制限、利用規約が変わるため、記事リポジトリへの自動保存までを無人化しない

そのため、生成APIの採用は保留し、現在のプロンプトコピーと手動アップロードを安全な標準フローとします。生成画像の採用判断と、実際のnoteへの画像挿入は人が行います。

## 画像タスクの状態

アプリの `image-status.json` では、プレースホルダーごとに次の判断を保存します。

| 状態 | 意味 | 次の操作 |
| --- | --- | --- |
| `pending` | まだ判断していない | `generate`、`provide`、`skip` のいずれかを選ぶ |
| `generate` | AI生成を使う | プロンプトをコピーし、生成結果を確認する |
| `provide` | 自分で写真・素材を用意する | 許諾と内容を確認して画像を登録する |
| `skip` | 画像を使わない | プレースホルダーを記事上で手動整理する |

生成結果を採用する場合は、内容・権利・記事との整合性を確認してから `provide` として画像登録します。採用しない場合は再生成、別素材への差し替え、または `skip` に戻します。登録後に問題が見つかった場合は、既存画像の置き換え、記事本文の差し戻し、Git履歴からの復元を行います。アプリは採用・却下を自動判定しません。

## 生成前後の確認項目

- 実在人物、来園者、キャスト、個人情報を入力・生成対象にしない
- 実在施設やアトラクションを扱う場合、生成画像を実際の写真と誤認させる説明を付けない
- 既存の公式画像、ロゴ、キャラクター画像を転載・模倣する用途に使わない
- 出力形式、長辺サイズ、容量を確認する。アプリの登録上限は5MBで、PNG/JPEG/WebP/GIFを受け付ける
- 生成サービスの料金、保存期間、商用利用条件を生成前に確認する

## 将来の候補

API連携が必要になった場合は、privateな記事リポジトリのGitHub Actionsを使う案を第一候補とします。プロンプトを手動でworkflow_dispatchへ渡し、Actions Secretに保存した鍵で生成し、レビュー用の画像を記事リポジトリへ保存する方式です。公開PWAから秘密鍵を受け取らず、生成失敗時に再実行できる点が条件です。

ただし、現時点では料金・サービス規約・生成物の権利確認を自動化できないため実装しません。採用判断は次の手順で検証できます。

1. 記事詳細で画像タスクを「AIで生成」にする
2. 生成用プロンプトをコピーし、利用するサービスへ手動で渡す
3. 出力の権利・内容・サイズ・形式を確認する
4. 「自分で用意」として5MB以下の画像を登録する
5. 本文への差し込み、画像プレビュー、同じタスクの置き換え、失敗後の再開を確認する

この手順で不足する場合だけ、privateリポジトリのActions方式を小さく試作します。静的PWAへAPIキーを追加する方式は採用しません。
